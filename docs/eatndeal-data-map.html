Overview

Architecture in one picture

web/ Express + EJS

Renders HTML pages. Never talks to Postgres. Holds the session (file-backed). Proxies every data fetch to api/ via callApi(req, METHOD, path). Injects customer_id from req.session.user.id on every authed call.

api/ Node + Knex

One controller per resource (Restaurants / Products / Categories / Offers / Auth / Address / Favourite). Knex builds SQL against wtw_eatndeal. Returns envelope {status, show, msg, data}. Validators in api/Validators/ gate every request.

wtw_eatndeal Postgres (live)

Shared with the legacy Yii2 eatndealclean POS/webordering. Marketplace reads the same product / branch / company tables (filtered by company.is_marketplace=1). New tables prefixed mp_ for things only the marketplace needs.

Browser PWA

EJS-rendered HTML. Client JS handles filters, the location sheet, the heart toggle, the cart pickers. Strict CSP (no inline JS/CSS); all assets served from the web origin.

Flow on every page load: Browser → web/ renders shell + calls api/ → api/ queries Postgres → envelope back to web/ → EJS renders, browser hydrates.

Product detail / Linked sub-groups

Linked sub-groups (option → nested group)

When an option in group A "opens" group B, there's a row in modifier_copy_details:

modifier_copy_details
  (group_id, modifier_option_id, link_group_id)

Dedup rule (same as legacy)

  • Collect every link_group_id referenced by the product's groups.
  • If a group's id is in that set, skip it from the top-level list — it will appear nested inside its parent option instead.
  • Without this, "Extra Toppings" would render twice (top level + nested) on Roll Back Pizza.

Rendering

  • Top-level group renders as a fieldset.
  • Nested sub-groups render as their own .pd-linked-section after the parent fieldset (not inside the grid).
  • Hidden until the parent option is checked. Switching to another option in the same parent group hides the previous sub-group and clears its picks (so they don't sneak into the price).
  • Validation skips a nested group while it's hidden.

Calculations / Distance

Distance — Haversine

Used everywhere "X.X km" appears on a card and to drive the delivery/pickup time estimates.

R = 6371                                  // Earth radius in km
dLat = toRad(lat2 − lat1)
dLng = toRad(lng2 − lng1)
a    = sin²(dLat/2) + cos(lat1)·cos(lat2)·sin²(dLng/2)
km   = 2R · asin(√a)
return round(km, 1)

Guards

  • Any missing / empty input ⇒ null (UI hides the chip).
  • Branch coords from branch.direction_latitude / direction_longitude.
  • User coords from session.userLocation.lat / lng.

Implementation: api/Helpers/distance.js · kmBetween().

Calculations / Delivery time

Delivery-time — exact legacy formula

Ports Commonquery::getAdvancedTime + displayRealOrderTime from the legacy eatndealclean webordering. No distance estimate, no fallback — if it's 0 in the DB, it shows 0.

total_min  =  baseTime
           +  valueTime
           +  volumeTime
           +  advanced_order_waiting_time_minute (stored on the order)

baseTime — branch cooking baseline

branch.delivery_waiting_time stored as D:H:M by admin. Parsed via:

parts = value.split(':')
3 parts → mins = parts[2] + parts[1] × 60   // last + middle×60
2 parts → mins = parts[1] + parts[0] × 60   // H:M
1 part  → mins = parts[0]                   // bare integer
0 / unset / null → null  (chip hides)

Matches legacy's date('s') + date('i') × 60 reading of strtotime — the admin's "minutes" land in the seconds slot, "hours" in the minutes slot.

valueTime — order-value rule

SELECT time_for_new_order_min FROM advance_order_waiting_time
 WHERE status=1 AND order_type=1
   AND service_type = 4               -- 4 = delivery, 1 = pickup
   AND branch_id = ? AND company_id = ?
   AND min_subtotal_amount <= cart_total
   AND max_subtotal_amount >= cart_total
 ORDER BY id ASC LIMIT 1

volumeTime — order-volume rule

count = SELECT COUNT(*) FROM orders
         WHERE branch_id = ? AND company_id = ?
           AND status IN (4, 5, 6, 10)
           AND DATE(created_at) = CURRENT_DATE

SELECT time_for_new_order_min FROM advance_order_waiting_time
 WHERE status=1 AND order_type=2
   AND service_type = 4
   AND min_subtotal_amount <= count
   AND max_subtotal_amount >= count
 LIMIT 1

Display — range, not single value

Listing pages don't have the cart total or live volume count, so they can't pick the EXACT row. Instead they show the worst-case range:

low  = baseTime
high = baseTime + sum(time_for_new_order_min)
       across every active advance row for this branch+service

Examples:
  base 20, no advance rules        → "20 min"
  base 20, value 15 + volume 20    → "20-55 min"
  base 0                            → null (chip hides)

One common entry point

Every page (list / detail / favourites / dishes) calls:

const times = await OrderTime.computeForBranches(branchRows);
//   times[branchId] = { delivery: "20-55 min" | null,
//                       pickup:   "30-60 min" | null }

One round trip for the whole page. Controllers just map branch_id → t.delivery / t.pickup; the parse + lookup + format dance lives ONLY in orderTime.js.

Implementation: api/Helpers/orderTime.js · computeForBranches(rows) · for checkout: totalMinutes(branch, cartTotal, mode).

Calculations / Open / Closed

Is the branch open right now?

// 1. Flag check
if (branch.open_as_usual === 0 AND branch.closed_until in the future)
   ⇒ CLOSED
else                              ⇒ pass

// 2. Clock check
open  = HH:MM from branch.start_time
close = HH:MM from branch.end_time
now   = local now

If open == close OR either missing  ⇒ OPEN (treat as always-on)
If open < close (e.g. 11:00–22:00) ⇒ OPEN iff now ∈ [open, close)
If open > close (e.g. 18:00–02:00) ⇒ OPEN iff now ≥ open OR now < close

Result used as the "Closed" label on cards + the "Open now" filter chip.

Implementation: marketplace.isOpenNow(branch) · helpers: isBranchOpen, clockMinutes.

Calculations / Price pick chain

Which price column does the UI show?

Many real rows haven't been re-priced for the marketplace yet — marketplace_price is often 0. Falling back to the next column means we always show something sensible.

pickPrice(product):
  candidates = [
    product.marketplace_price,       // explicit marketplace listing
    product.online_platform_price,   // existing webordering price
    product.price_after_tax,         // in-store price
  ]
  for v in candidates:
      n = Number(v)
      if isFinite(n) AND n > 0:
          return round(n, 2)
  return 0

Same chain feeds: card prices, product page basePrice, price-bucket filter (low ≤ £6 / mid £6-12 / high > £12).

Implementation: marketplace.pickPrice(row).

Calculations / has_offer

"Has offer" — 4-way EXISTS

One SQL flag per restaurant card, true if ANY of the four conditions holds:

EXISTS (SELECT 1 FROM store_offer_banner_table sob
        WHERE sob.company_id = c.id
          AND sob.publish_online = 1
          AND (sob.start_date IS NULL OR sob.start_date <= CURRENT_DATE)
          AND (sob.end_date   IS NULL OR sob.end_date   >= CURRENT_DATE))
OR EXISTS (SELECT 1 FROM coupons cp
        WHERE cp.company_id = c.id
          AND cp.is_active = 1
          AND cp.platform IN (1, 2)
          AND (cp.expiry_date IS NULL OR cp.expiry_date >= CURRENT_DATE))
OR EXISTS (SELECT 1 FROM discounts d
        WHERE d.company_id = c.id
          AND d.status = 1
          AND (d.start_date IS NULL OR d.start_date <= CURRENT_DATE)
          AND (d.end_date   IS NULL OR d.end_date   >= CURRENT_DATE))
OR EXISTS (SELECT 1 FROM products op
        WHERE op.company_id = c.id
          AND op.show_marketplace = 1
          AND op.status = '1'
          AND op.offer IS NOT NULL AND op.offer <> '' AND op.offer <> '0')

The badge label + count come from a second pass (offerSummaries) — the largest %-off wins, else "Free delivery", else "£N OFF", else "Offer".

Calculations / Rating

Restaurant rating

avg_rating  = ROUND(AVG(rr.rating), 1)
rating_count = COUNT(*)

  FROM review_rating rr
 WHERE rr.company_id = c.id
   AND rr.publish_online = 1

Display

  • Card: shows avg_rating; falls back to 4.4 when no published rows (avoids zero stars on a brand-new restaurant).
  • Detail page: also shows "(N+)" rating count when count > 0.
  • Filter "Rating ≥ 4.0" uses the real avg — fallback 4.4 still applies if no rows, which means an unrated restaurant can pass a 4.0+ filter.
Calculations / Veg classification

Veg / Non-veg / Pure-veg

Per product

products.veg_non_veg: 1 ⇒ veg · 2 ⇒ non-veg · 3 ⇒ egg/other. We treat 1 only as veg.

Per restaurant card

has_veg     = EXISTS(product WHERE company_id = c.id AND veg_non_veg = 1
                              AND show_marketplace = 1 AND status='1')
has_non_veg = EXISTS(product WHERE company_id = c.id AND veg_non_veg <> 1
                              AND show_marketplace = 1 AND status='1')

vegType =
   'pure-veg' if has_veg AND NOT has_non_veg
   'non-veg'  if has_non_veg                  (one non-veg ⇒ non-veg overall,
                                               FSSAI-style)
   null       if neither (no marketplace products yet)

The "Pure veg only" filter keeps only pure-veg restaurants.
